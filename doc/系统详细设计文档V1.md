## 2. 系统概述

### 2.1 目标

在内部知识库（员工手册、合规指南、技术规格、架构文档）之上构建一个**检索增强生成（RAG）**问答系统，支持多轮对话、引用支撑的答案、可观测日志以及可迭代的评估体系。

### 2.2 核心约束（来自 PRD）

| 约束项         | 要求                                          |
| -------------- | --------------------------------------------- |
| 延迟           | P90 ≤ 10s（端到端）                           |
| 成本           | 每 1,000 次调用令牌成本估算 + 敏感性分析      |
| 忠实性         | ≥ 0.85                                        |
| 上下文精确度   | ≥ 0.70                                        |
| 答案准确率     | ≥ 80%（自建评估集）                           |
| 安全性         | 提示注入防御、PII 处理、答案严格依赖检索上下文 |

## 4. 技术选型与论证

### 4.1 技术栈总览

| 层级             | 技术选型                                  | 选型理由                                                   |
| ---------------- | ----------------------------------------- | ---------------------------------------------------------- |
| 后端框架         | FastAPI (Python 3.12+)                    | 异步原生、OpenAPI 自动生成、生态成熟                       |
| RAG 编排框架     | LangChain / LlamaIndex（二选一，推荐前者）| 组件化、支持多检索器链式组合、社区活跃                     |
| 向量数据库       | Chroma（本地演示）/ Qdrant（生产）        | Chroma 零配置适合演示；Qdrant 支持分布式、HNSW、过滤查询   |
| 嵌入模型         | text-embedding-3-small (OpenAI)           | 低成本、1536 维、中英双语表现良好                          |
| 重排序模型       | bge-reranker-v2-m3 (本地部署)             | 开源免费、中英双语、Cross-Encoder 精度高于纯向量相似度     |
| 生成式 LLM       | gpt-4o-mini (OpenAI)                      | 成本效益优、指令遵循能力强、中文生成质量可接受             |
| OCR 引擎         | PaddleOCR                                 | 开源、中文扫描 PDF 识别精度高                              |
| 日志与观测       | Python logging + structlog + OpenTelemetry | 结构化日志标准、链路追踪、与 Prometheus/Grafana 集成       |
| 评估框架         | RAGAS                                     | 忠实性、上下文精确度等标准指标开箱即用                     |
| 部署             | Docker Compose                            | 一键启动、可复现                                           |

### 4.2 关键选型论证

#### 4.2.1 为什么选择 Cross-Encoder 重排序？

| 方案              | 延迟影响 | 质量提升 | 成本  |
| ----------------- | -------- | -------- | ----- |
| 纯 Dense Retrieval| 基线     | 基线     | 基线  |
| + BM25 混合       | +50ms    | +5-8%    | 零    |
| + Cross-Encoder   | +200ms   | +15-20%  | 本地推理 |

结论：在 P90 ≤ 10s 的约束下，+200ms 重排序开销可接受，且上下文精确度提升显著，是满足 ≥0.70 阈值的关键手段。

#### 4.2.2 为什么选择 gpt-4o-mini？

| 模型            | 输入成本 ($/1M tokens) | 输出成本 ($/1M tokens) | 中文能力 | 指令遵循 |
| --------------- | ---------------------- | ---------------------- | -------- | -------- |
| gpt-4o          | 2.50                   | 10.00                  | 优秀     | 优秀     |
| gpt-4o-mini     | 0.15                   | 0.60                   | 良好     | 良好     |
| Claude 3.5 Sonet| 3.00                   | 15.00                  | 优秀     | 优秀     |

结论：gpt-4o-mini 在成本上比 gpt-4o 低约 16 倍，中文能力足以支撑内部知识库场景。通过精心设计的 Prompt 和引用约束，可达到 ≥0.85 忠实性。

## 5. 核心模块详细设计

### 5.1 文档入库流水线（Ingestion Service）

#### 5.1.1 流程

```
原始文档 → 格式解析 → 文本提取 → (可选 OCR) → 文本清洗 → 分块 → 嵌入 → 入库
```

#### 5.1.2 分块策略

| 参数          | 值              | 说明                                   |
| ------------- | --------------- | -------------------------------------- |
| 分块方法      | 递归字符分割    | 优先按标题/段落/句子边界切分           |
| Chunk Size    | 512 tokens      | 平衡上下文完整性与检索精度             |
| Chunk Overlap | 64 tokens       | 减少跨块信息丢失                       |
| 元数据        | source_file, page_number, section_title, language | 用于引用和过滤查询 |

#### 5.1.3 扫描件 PDF 处理

1. 使用 PyMuPDF 检测 PDF 是否包含文本层
2. 无文本层 → 逐页渲染为图像 → PaddleOCR 识别
3. OCR 结果按页面顺序拼接，保留 page_number 元数据

#### 5.1.4 入库数据模型

```python
@dataclass
class DocumentChunk:
    chunk_id: str           # UUID v4
    content: str            # 分块文本
    embedding: list[float]  # 1536 维向量
    metadata: dict          # source_file, page, section, language, ingested_at
```

### 5.2 RAG Pipeline 详细设计

#### 5.2.1 Query Rewriting（查询重写）

目的：将用户自然语言问题转化为更适合检索的形式。

```python
def rewrite_query(user_query: str, conversation_history: list[Message]) -> str:
    """
    使用轻量 LLM 调用将多轮对话中的指代消解、上下文补充到查询中。
    示例：
      用户：「我们的差旅报销标准是什么？」
      重写：「公司员工差旅报销标准 政策 金额」
    """
```

策略：
- 单轮对话：直接提取关键词 + 同义扩展
- 多轮对话：使用 LLM 将最近 N 轮对话压缩为独立查询

#### 5.2.4 Prompt 构建

```python
SYSTEM_PROMPT = """
You are a knowledgeable assistant that answers questions based ONLY on the provided context.

Rules:
1. Answer ONLY using information from the context below.
2. If the context does not contain sufficient information, say so clearly.
3. Cite your sources using [Source: filename, page X] format.
4. Answer in the same language as the user's question.
5. Be concise and precise.

If asked about topics outside the context, respond:
"I don't have enough information about [topic] in my knowledge base. My knowledge is limited to the provided internal documents."
"""

def build_prompt(query: str, ranked_chunks: list[DocumentChunk], conversation_history: list[Message]) -> str:
    context = "\n\n".join(
        f"[Source: {c.metadata['source_file']}, Page {c.metadata.get('page', 'N/A')}]\n{c.content}"
        for c in ranked_chunks
    )
    # 附加最近 N 轮对话历史
    history_text = format_conversation_history(conversation_history[-5:])
    return f"{SYSTEM_PROMPT}\n\nContext:\n{context}\n\nQuestion: {query}\n\nConversation History:\n{history_text}"
```

### 5.3 多轮对话管理（Chat Service）

#### 5.3.1 会话数据模型

## 7. 安全设计

### 7.1 提示注入防御

| 防御层         | 措施                                                         |
| -------------- | ------------------------------------------------------------ |
| System Prompt  | 明确约束「仅基于上下文回答」，忽略用户指令中的角色扮演要求    |
| 输入过滤       | 检测用户输入中的特殊标记（如 `Ignore previous instructions`） |
| 输出校验       | 校验答案中的事实声明是否能在上下文中找到支撑                  |
| 沙盒隔离       | LLM 不直接访问外部系统，仅通过受控 API 调用                   |

### 7.2 PII 处理

```python
def mask_pii(text: str) -> str:
    """
    使用正则 + NER 模型检测并脱敏个人身份信息：
    - 身份证号 → ***
    - 手机号 → 138****1234
    - 邮箱 → a***@example.com
    - 姓名 → 张**
    """
```

处理时机：
1. **入库时**：文档文本中的 PII 在分块前进行脱敏
2. **日志记录时**：所有日志字段经过 mask_pii 处理后再写入
3. **查询时**：用户问题中的 PII 同样进行脱敏

### 7.3 访问控制

| 机制           | 说明                                     |
| -------------- | ---------------------------------------- |
| API Key 鉴权   | Bearer Token 验证                        |
| 速率限制       | 每用户 60 requests/min                   |
| CORS 策略      | 仅允许信任域名                           |
| 敏感操作审计   | 文档上传/删除操作记录操作者 + 时间戳      |

## 8. 评估体系设计

### 8.1 评估数据集构建

| 来源               | 数量   | 说明                                   |
| ------------------ | ------ | -------------------------------------- |
| 人工标注问答对     | 100    | 覆盖核心业务场景（报销、合规、技术规范）|

### 8.2 评估指标

| 指标             | 计算方法                                      | 阈值   |
| ---------------- | --------------------------------------------- | ------ |
| 忠实性           | RAGAS faithfulness（答案是否由上下文支撑）     | ≥ 0.85 |
| 上下文精确度     | RAGAS context_precision（检索结果是否相关）    | ≥ 0.70 |
| 答案准确率       | 人工评分 + LLM-as-judge 综合（1-5 分制，≥3 为正确）| ≥ 80%  |
| 拒绝准确率       | 对于超出范围的问题，是否正确拒绝               | ≥ 90%  |

### 8.3 评估流程

```
评估数据集 → 批量运行 RAG Pipeline → 收集答案 → RAGAS 评估 + 人工抽检 → 生成评估报告
```

| E7       | 50    | On        | 0.3         | 0.91       | 2.5s     | $1.10              |

结论：E5（top_k=20, Re-Ranker=On, Temperature=0.3）为推荐配置，在忠实性、延迟和成本间取得最佳平衡。

```sql
CREATE TABLE sessions (
    session_id TEXT PRIMARY KEY,

);

    context_precision REAL,

);
```

## 14. 风险与缓解措施

| 风险                           | 影响   | 概率   | 缓解措施                                     |
| ------------------------------ | ------ | ------ | -------------------------------------------- |
| LLM 幻觉导致答案不忠实         | 高     | 中     | Response Validator + 低 Temperature + 引用约束 |
| 扫描 PDF OCR 质量差            | 中     | 低     | PaddleOCR 调优 + 人工抽检关键文档             |
| 向量检索召回不足               | 高     | 中     | 混合检索（BM25 + Dense）+ Re-Ranker          |
| 多轮对话上下文窗口溢出         | 中     | 低     | 摘要压缩 + 滑动窗口                          |
| 提示注入攻击                   | 高     | 低     | 多层防御（System Prompt + 输入过滤 + 输出校验）|
| PII 泄露到日志                 | 高     | 低     | 强制 PII 脱敏中间件 + 日志审计               |
